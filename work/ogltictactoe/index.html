The first game most people make is usually the classic Tic-Tac-Toe. When I started my first Tic-Tac-Toe game in Python, I had no idea what to do, because it was so conceptually different from what I am used to. Input comes from the hands, not from the keyboard. Typing '4' or '5' to take the center piece or even '2 2', makes no sense for an actual game. Because I had not learned about PyGame yet, I looked back at what input I actually knew, and took a stab at OpenGL.

My first encounter with OpenGL was from my Counter-Strike days, where my tinkering with the in-game configurations and macros made me notice the subtle D3D and OpenGL options. I never really understood these terms, except that they had something to do with graphics, so I tried to fix slightly.

Process

Using a finite state machine, as I had learned in the previous term, I wrote out a basic layout of how the state progression works in a game.

STARTMENU -> GAME -> RESULTS -> STARTMENU

Something like that, and progression through the states became simply reading a bunch of if statements. It was really cool to see something like that actually work.

The rest of the implementation went by pretty normally. I noticed something that made checking the grid for winning a lot easier. If I created a vector that stored the 8 3-number combinations that need to be checked, I could just iterate through the vector to pass through the entire grid, and if I wanted to randomize things a bit, so for my A.I., I could just randomize that vector. This was a refreshing take for checking for a Tic-Tac-Toe winner.

After I had the board set-up fully, and an actual game available, creating the three A.I. levels was the fun part. Using wikipedia's Tic-Tac-Toe strategy and randomizing it a bit, along with some other custom pieces, I created a Level 3 A.I. that never loses, even if it goes second. Toning down the difficulty simply required stripping the more advanced move algorithms, while the Level 1 A.I. played randomly unless it could win or block. Simple, but fun to code.

DirectX v. OpenGL

There is actually a lot of debate over OpenGL versus DirectX. Because I saw them around a lot in Counter-Strike, it really interested me to take a look into the craziness that is the WinAPI.

I encountered this post, Why Do Game Developers Prefer Windows, which was a really great read, not to mention a good look into the history of OpenGL and the history of WinAPI.

I also saw this post, D3D vs OpenGL, which looks like the general consensus of programmers in that field. While OpenGL is more 'open', and therefore can support better and more advanced features, because Microsoft pushed DirectX so heavily, we are stuck with the good, but not best.
